《天涯明月刀》AI玩家控制系统 PRD
v2.0（按阶段落地版）

状态：执行到一半
当前进展：第一阶段已收敛为单对话框原型；当前可验证文字 / 语音输入、思考链展示与模拟动作规划。截图识别原型暂未接入主对话链路。

一、项目目标

本系统的目标，是构建一个“可被观看的 AI 实体玩家”。
它不仅要能够理解主播的指令、形成可展示的思考过程、输出带人格的表达，还要在条件成熟时进一步接入真实游戏控制能力，完成 NPC 互动、社会行为与剧情推进。

需要特别强调的是，本系统的最终目标不是“用户每说一句，AI 回一轮计划”的被动助手，而是一个近实时运行的全自动游戏 Agent。

该 Agent 的最终工作方式应为：

1. 用户只需用自然语言下达高层目标，例如“你去和 AAA 搞好关系”。
2. AI 持续近实时获取游戏画面，自动判断当前所处场景、任务状态、风险和机会。
3. AI 自动执行移动、寻路、交互、接任务、做任务、升级、探索地图、和 NPC 互动等操作。
4. 当用户没有输入时，AI 保持自主探索与自主推进。
5. 当用户有新的输入时，AI 必须立即提高该指令优先级，打断或重排当前自主行为，优先执行用户目标。
6. 因此，视觉感知必须是持续运行的近实时链路，而不是“每轮对话附带一次截图”的低频补充功能。

本项目当前的现实约束是：

1. 现阶段主要开发环境为 macOS。
2. 目标游戏无官方 API。
3. 类似按键精灵或 Windows 底层输入模拟的能力，当前不适合在 macOS 上作为第一阶段核心交付。

本项目的最终交付要求是：

1. 成品最终运行环境为一台固定的 Windows 主机，而不是面向任意用户机器发布安装包。
2. 系统允许在该 Windows 主机上预先安装必要运行环境，但部署步骤必须可控、可复现、可文档化。
3. 因此，技术选型应优先考虑“在固定 Windows 主机上最快跑稳”，而不是优先考虑桌面封装或一键安装分发。

因此，本项目不再以“第一天就完成真实自动操控”为目标，而采用分阶段建设策略：

第一阶段，先完成“AI 可被观看、可被指挥、可展示思考、可输出人格”的核心体验。
第二阶段，补齐视觉识别与半自动验证能力。
第三阶段，再接入 Windows 平台下的真实输入控制能力，并把系统升级为持续运行的全自动 Agent，形成完整闭环。

二、产品定位

本系统是一个面向直播和内容创作场景的 AI 互动控制系统。
它的核心价值不是竞技强度，而是“可观看性”与“人格化”。

对外呈现上，它应该像一个拥有独立性格、会误解、会权衡、会解释自己行为的 AI 玩家。
对内实现上，它应该是一个可拆分的系统：

- 平台无关部分：语义理解、思考链生成、行为规划、日志、状态管理、前端展示、人格表达。
- 平台相关部分：屏幕采集、OCR、模板识别、鼠标键盘注入、系统级音频或输入控制。

从产品形态上看，它最终应更接近“会自己玩、会自己找事做、会被主播临时改命令的自主代理”，而不是一个只能等待用户逐轮驱动的问答机器人。

三、阶段性目标

阶段一：macOS 可运行的最小产品（MVP）

目标：

- 让主播可以通过文字方式向 AI 下达指令。
- 让主播可以通过浏览器语音把口语指令写入同一个输入框。
- 让 AI 生成结构化思考链，并在客户端实时展示。
- 让 AI 输出行为规划结果，而不是直接执行真实鼠标点击。
- 让客户端以单对话框形态承载输入、回复、思考链和动作建议。
- 让系统的“人格表现”和“直播观感”先被验证。

本阶段不追求：

- 真实控制游戏。
- 完整语音克隆。
- 高精度 OCR 或跨场景泛化。

阶段二：视觉识别与半实物验证

目标：

- 接入近实时截图、实时截屏或录屏帧输入。
- 识别当前界面类型、NPC 名称、基础互动选项、异常提示。
- 验证“语义理解 -> 视觉判断 -> 行为规划”的链路是否稳定。
- 验证持续感知循环是否能够稳定支持自动规划。
- 执行层依然允许保留为“模拟执行器”，只输出下一步动作建议。

阶段三：Windows 真实控制版本

目标：

- 接入 Windows 原生输入模拟能力。
- 实现鼠标、键盘、行为节奏随机化与中断控制。
- 将模拟执行器切换为真实执行器。
- 形成“持续感知 -> 自主规划 -> 真实执行 -> 反馈复检 -> 异常恢复”的完整自动化闭环。
- 实现“空闲时自主推进、接到用户命令时优先改道执行”的双模式运行。

四、总体架构

系统采用分层架构，而非单体脚本式实现。

1. 交互层

负责接收主播输入，并将系统状态反馈给主播。
第一阶段建议以 Web 对话框作为客户端形态，即一个本地运行的 HTML 页面。

交互层包含：

- 文字输入
- 浏览器语音输入入口
- AI 回复展示
- 思考链展示
- 动作建议展示
- 截图能力的接入状态说明

需要注意：交互层只是“插手和观看”的入口，不是主驱动源。最终真正驱动系统持续运行的是后台 Agent 主循环，而不是用户每次手动提交消息。

2. 决策层

负责把自然语言指令转化为结构化任务。
该层由大模型驱动，但输出必须是受控结构，而不是自由散文。

建议输出结构至少包括：

- 主播意图理解
- 当前环境判断
- 候选策略
- 风险评估
- 当前选择
- 下一步动作计划

3. 行为规划层

负责把抽象目标转化为可执行步骤。
例如，“和 NPC 混熟”不会直接变成点击，而应先变成：

- 确认当前是否已处于对话范围
- 判断 NPC 是否可互动
- 判断背包内是否有礼物
- 若无礼物，转为闲聊或其他互动路径
- 输出下一步操作建议

这一层的输出应该是标准化 Action 列表，而不是自然语言段落。

4. 感知层

负责理解当前游戏或模拟环境的状态。
这一层不应和具体平台强耦合，而应通过适配器接入。

感知层输入形式可以按阶段演进：

- 阶段一：先不把截图识别挂入主对话链路，优先验证纯对话交互与规划质量
- 阶段二：接入近实时截图、OCR 和模板匹配，建立持续感知循环
- 阶段三：接入稳定的游戏窗口采集方案

5. 执行层

负责执行 Action 列表。
该层必须抽象为接口，以便不同平台替换实现。

建议至少有两种执行器：

- MockExecutor：只展示“将要执行什么”，不真实点击
- WindowsExecutor：在 Windows 环境下执行真实鼠标键盘输入

最终执行层不是只执行用户点名的一小段操作，而是要能持续驱动角色活动，包括：

- 自主探索地图
- 自动接取与推进任务
- 自动和 NPC 互动
- 自动战斗或规避风险
- 自动恢复到主循环

6. 表达层

负责把 AI 的思考、态度和选择转化为对外可感知的表达。
第一阶段可先用文本展示与普通 TTS。
后续再升级为定制语音人格或语音克隆方案。

五、第一阶段客户端形态

第一阶段客户端不再定义为 Windows 独立程序，而定义为：

“本地 Web 对话框 + 本地服务”

其中：

- Web 对话框负责展示和交互，可以是一个 HTML 页面。
- 本地服务负责调用模型、维护状态、生成思考链和行为计划。
- 截图识别能力在本阶段保留为独立原型，不接入当前主对话流程。

这样设计的原因是：

1. 开发速度快，适合快速迭代交互体验。
2. 单对话框更接近直播使用习惯，减少主播操作负担。
3. 当前不再以桌面封装为优先目标，可直接保留为本地服务 + 浏览器对话框形态。
4. 前端与执行层解耦，后期切换到 Windows 主机运行时无需重写大部分业务逻辑。

但这不代表最终产品只停留在对话框形态。对话框只是开发阶段的人类观察窗，最终核心仍然是后台自动运行的 Agent 主循环。

六、最终部署形态与运行要求

最终交付目标不再是 Windows 安装包，而是在一台固定的 Windows 主机上部署并稳定运行。

具体要求如下：

1. 系统默认部署在一台固定的 Windows 主机上，运行环境、分辨率、UI 缩放、语言和游戏版本都应尽量固定。
2. 允许在该主机上手动安装 Node、OCR 依赖、视觉库或其他必要运行组件，但必须形成明确的部署文档。
3. 运行时可以保留“本地服务 + 浏览器对话框”形态，不要求封装为桌面安装包。
4. 若系统需要本地服务、模型调用配置、截图能力或输入控制模块，应在该主机上按统一流程部署，而不是面向不特定机器泛化分发。
5. 技术选型时应优先考虑 Windows 主机上的识别速度、点击稳定性和调试便利性，而不是安装包体积或跨机器分发成本。
6. 后续如确有需要，可在项目后期再评估是否封装为桌面程序，但这不再是当前阶段的前置约束。

七、第一阶段核心功能

1. 指令输入

主播通过文本或浏览器语音输入指令，例如：

- 帮我和这个 NPC 混熟
- 赚钱
- 给他点颜色看看
- 别太老实

系统接收后，进入一轮新的任务状态。

2. 思考链展示

系统生成受控、可读、适合直播展示的思考链。
思考链不是无限展开，而应控制长度、节奏和风格。

建议思考链分为三段：

- 我理解你想要什么
- 我判断当前环境是什么
- 我准备怎么做

示例：

“目标不是单次互动，而是建立关系优势。”
“当前最重要的是先确认对方是否能进入高频互动状态。”
“如果资源不足，我会先走低成本接近策略。”

3. 对话内结构化展示

所有关键信息都展示在同一个对话框内，而不是拆成多个控制面板。
每轮 AI 回复至少要在对话中可见：

- 当前环境判断
- 当前阶段能力说明
- 思考链条
- 风险等级
- 下一步动作建议

4. 自动感知输入

在没有真实游戏控制能力时，系统仍然需要环境输入。
但考虑到当前主要开发环境仍是 macOS，且最终执行环境是 Windows，第一阶段暂不把截图识别挂入主对话链路。

今天在 macOS 上可验证的方式是：

- 用户发送一句话
- 浏览器语音或文本写入同一个输入框
- 本地服务根据文本指令生成规划
- 在对话框里返回回复、思考链和动作建议

5. 模拟执行器

第一阶段不执行真实点击，只输出诸如以下内容：

- 打开互动菜单
- 选择“交谈”
- 检查是否存在“赠礼”选项
- 若存在，则优先选择低风险礼物

这部分既用于调试，也用于验证行为规划质量。

八、最终 Agent 运行原则

最终版本必须遵守以下运行原则：

1. 默认自主运行：在用户没有输入时，AI 不应停住等待，而应持续寻找可推进的目标。
2. 用户命令优先：只要用户发出新的自然语言指令，AI 必须立即提升该目标优先级。
3. 可中断重规划：当前动作链必须支持被打断、重排和重新规划。
4. 持续感知驱动：系统依靠近实时截图和状态识别运行，而不是低频单次截图。
5. 任务与探索并存：AI 既能执行明确任务，也能在空闲时自己刷任务、升级、跑图、社交或做资源积累。
6. 自主行为可被观看：即使用户没有输入，AI 也应持续产出可解释、可展示的行为过程。

九、语义理解与行为规划要求

语义层必须允许一定的“人格化偏差”，即 AI 可以有风格化解读，但不能脱离动作边界。

例如：

- “帮我混熟”可以被解释为高频互动、送礼或制造强记忆点。
- “给他点颜色看看”可以被解释为威慑、试探、挑衅，但不能无限制发散。

为保证可控性，建议建立受限动作集合。

动作集合示例：

- talk
- gift
- inspect
- trade
- threaten
- steal
- strike
- escape
- wait

所有模型输出都应先映射到受限动作集合，再进入执行器。

十、视觉识别设计

视觉识别模块目标不是“一开始就全自动”，而是逐步提高自动化程度。

第一阶段：

- 截图识别保留为独立验证能力
- 不接入当前主对话界面

第二阶段：

- 近实时截图采集
- OCR 识别界面文字
- 模板匹配识别固定 UI
- 提取 NPC 名称、菜单选项、秩序提示、通缉提示等信息
- 为自主运行循环提供持续状态输入

第三阶段：

- 与实时采集方案整合
- 为执行层提供稳定、低延迟的状态反馈
- 进入后台持续运行模式，而不是仅在用户发言后触发

十一、语音与人格模块

人格模块是本系统的内容价值核心之一。
但在工程排期上，应按复杂度分层处理。

第一阶段：

- 文本人格先行
- 可选基础 TTS 播报

第二阶段：

- 优化人设口吻
- 增强思考链节奏与反应速度

第三阶段：

- 接入特定人格语音方案
- 实现更稳定的角色化表达

十二、异常处理与安全控制

无论是否进入真实执行阶段，系统都必须具备中断与恢复机制。

至少包括：

- 模型输出不合法时停止执行
- 场景识别置信度不足时转人工确认
- 执行器报错时进入暂停状态
- 急停后清空当前动作队列

在真实控制版本中，还需补充：

- 异常界面回退
- 通缉或高风险状态保护
- 人工接管优先

十三、推荐技术实现方向

第一阶段推荐架构：

- 前端：HTML + CSS + JavaScript，或基于轻量前端框架
- 后端：本地服务，负责模型调用、状态机、日志、计划生成
- 通信：HTTP + WebSocket 或 SSE，用于实时推送思考链和状态
- 感知层：先保留为独立原型，不进入当前主对话闭环
- 执行层：MockExecutor

第二阶段新增：

- 近实时截图循环
- OCR 服务
- 模板匹配模块
- 截图采集模块

第三阶段新增：

- Windows 输入模拟模块
- 真实执行器
- Agent 主循环调度器
- 用户命令优先级仲裁器
- 更完整的异常恢复机制

补充约束：

- 正式交付链路应面向固定 Windows 主机部署，而不是通用安装包分发。
- 如需使用 Web 技术栈，应优先考虑 Windows 主机上的调试便利性和运行稳定性。
- 配置项可以保留为可编辑文件或启动前配置步骤，但必须有清晰的部署说明与恢复流程。

十四、当前版本结论

本项目当前最合理的推进路径，不是直接追求“按键精灵式全自动控制”，而是先完成一个可运行、可展示、可调试、可验证产品价值的 AI 对话框。

因此，当前版本的核心结论是：

1. 客户端可以是一个 HTML 页面。
2. 更准确地说，第一阶段应采用“本地 Web 对话框 + 本地服务”方案。
3. 真实游戏控制能力不应阻塞第一阶段开发。
4. 第一阶段先验证对话体验、人格表现、思考链展示和行为规划质量。
5. 但最终产品目标始终是近实时感知驱动的全自动 Agent，而不是停留在“逐轮聊天规划”。
6. 后续在 Windows 环境下补齐持续感知、执行器和主循环调度后，系统才算升级为完整控制系统。
7. 当前正式目标是“在固定 Windows 主机上稳定运行”，而不是面向任意机器发布 `.exe` 安装包。
