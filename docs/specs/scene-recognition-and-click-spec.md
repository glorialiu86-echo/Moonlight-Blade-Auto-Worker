# 《天涯明月刀》PC 端场景识别与点击规格

状态：未执行

## 目的

本规格用于指导《灵枢绘世》AI 玩家控制系统第二、三阶段的真实界面识别与点击实现。

目标不是构建一个通用电脑视觉代理，而是针对《天涯明月刀》PC 端固定 UI 环境，建立一套可控、可验证、可扩展的场景识别与点击规则体系。

本规格严格基于公开的《天涯明月刀》PC 端界面资料与交互说明做前期规划，后续实现时必须再以目标机器上的真实游戏截图样本进行校准。

## 设计原则

1. 不采用“全屏截图交给大模型直接输出点击坐标”的方案。
2. 主链路采用“模板匹配 + 局部 OCR + 少量规则”。
3. 大模型仅用于补充判断、场景总结、歧义解释与策略规划，不直接作为高频点击定位主链路。
4. 点击目标必须来自预定义场景文件，而不是由模型自由生成。
5. 每次点击后必须立刻复检场景是否切换，失败则停止或回退。
6. 实现时必须固定分辨率、固定 UI 缩放、固定语言、固定窗口模式，否则不进入自动化主链路。
7. 最终运行形态应面向近实时持续感知，而不是“用户发一句话才分析一张图”。

## 最终自动化目标

本规格服务的不是一个“单次问答式截图分析器”，而是一个会持续运行的全自动 Agent。

这意味着视觉系统最终要承担以下职责：

1. 近实时持续采集当前游戏窗口。
2. 把场景识别结果稳定提供给自主规划器。
3. 在用户没有输入时，支撑 AI 自主跑图、任务推进、社交互动和资源积累。
4. 在用户发出高优先级自然语言命令时，快速为新目标提供视觉反馈。
5. 支撑动作中断、回退、复检和重新规划。

因此，本规格中的每个场景都不应只考虑“能否识别一次”，还必须考虑“能否被持续轮询、持续复检和低延迟切换”。

## 优先场景

第一批只实现以下 3 个场景：

1. `npc_interact`
2. `shop`
3. `bag`

选择理由：

- 都是高频基础交互场景。
- 都存在公开资料可参考的稳定 UI 结构。
- 动作链条短，便于快速打通“识别 -> 决策 -> 点击 -> 复检”闭环。

## 场景 1：`npc_interact`

### 目标

识别“当前已靠近 NPC，且可以发起交互或对话”的状态。

### 公开依据

- 靠近 NPC 时会出现快捷交互提示，可按 `F` 发起对话。
- 左键可选中 NPC。
- 右键点击头像可执行更多交互。

来源：

- 新浪《天涯明月刀操作模式详解 新手入门必看》  
  https://games.sina.com.cn/o/z/wuxia/2015-07-09/fxewnie5378673.shtml

### 场景锚点

`anchor_f_prompt`

- 类型：OCR + 文本匹配
- 用途：识别快捷交互提示
- 关键词：`F`、`对话`、`交互`
- 优先级：最高

`anchor_target_panel`

- 类型：模板匹配 + OCR
- 用途：识别已选中目标后的名字/等级/头像区
- 读取内容：NPC 名称、目标信息

`anchor_world_hud`

- 类型：模板匹配
- 用途：确认当前仍处于主世界 HUD，而非已进入背包、商店或其他弹窗

### OCR 区域

`roi_target_name`

- 内容：NPC 名称

`roi_f_prompt`

- 内容：快捷交互提示文字

`roi_target_meta`

- 内容：等级、身份、附加目标信息

### 可点击目标

`click_npc_body`

- 用于首次选中 NPC

`click_target_avatar`

- 用于右键交互链路前的头像定位

`key_f_interact`

- 用于触发快捷交互
- 在该场景中应优先于鼠标点击

### 成功判定

`success_target_locked`

- NPC 名称稳定显示至少 2 帧

`success_dialog_opened`

- 对话框、交互菜单或新场景锚点出现

### 失败判定

`fail_no_prompt`

- 多次重试后仍未出现快捷交互提示

`fail_target_lost`

- NPC 名称框消失或目标丢失

### 推荐动作顺序

1. 如果识别到 `anchor_f_prompt`，优先按 `F`。
2. 如果没有 `F`，但存在目标信息区，则点击 NPC 本体或头像。
3. 如果两者都不存在，不应尝试点击 UI，应判定为“尚未接近成功”。

## 场景 2：`shop`

### 目标

识别“当前已经进入 NPC 商店，可执行购买或出售”。

### 公开依据

- 商店支持从商店物品到背包的购买流程。
- 支持从背包到商店的出售流程。
- 说明商店和背包区域会共同出现在交易界面中。

来源：

- 新浪《天涯明月刀操作模式详解 新手入门必看》  
  https://games.sina.com.cn/o/z/wuxia/2015-07-09/fxewnie5378673.shtml

### 场景锚点

`anchor_shop_title`

- 类型：OCR + 标题条模板
- 用途：识别商店标题或商店名

`anchor_shop_goods_grid`

- 类型：模板匹配
- 用途：识别商品列表区域

`anchor_shop_bag_grid`

- 类型：模板匹配
- 用途：识别背包区域

`anchor_buy_tabs`

- 类型：OCR + 模板匹配
- 用途：识别购买/出售/分类页签

### OCR 区域

`roi_shop_title`

- 内容：商店标题或 NPC 店名

`roi_goods_names`

- 内容：商品名列表

`roi_goods_prices`

- 内容：商品价格

`roi_tabs`

- 内容：页签名称

`roi_bag_item_name`

- 内容：背包选中物品名

### 可点击目标

`click_first_goods_slot`

- 商品列表第一格

`click_buy_tab`

- 切换到购买页

`click_sell_tab`

- 切换到出售页

`click_confirm_buy`

- 确认购买

`click_close_shop`

- 关闭商店

### 成功判定

`success_shop_open`

- 商店标题 + 商品区 + 背包区同时存在

`success_buy_feedback`

- 背包数量变化、货币变化或购买反馈出现

`success_sell_feedback`

- 背包物品数量减少或货币变化

### 失败判定

`fail_title_only`

- 仅检测到标题，没有检测到商品区或背包区

`fail_no_grid`

- 商品区或背包区缺失

`fail_warning_popup`

- 出现超出预期的确认框或风险弹窗

### 推荐动作顺序

1. 先确认标题、页签、商品区和背包区同时成立。
2. 再读取商品名和价格，不要直接点击购买。
3. 只有在商品区和背包区都稳定存在时，才允许执行买卖动作。

## 场景 3：`bag`

### 目标

识别“当前背包已打开，可查看、整理、拆分和选中物品”。

### 公开依据

- 背包界面存在 `拆分`、`整理` 等稳定操作按钮。
- 背包支持拖拽与装备整理，说明格子区是稳定主结构。

来源：

- 新浪《天涯明月刀操作模式详解 新手入门必看》  
  https://games.sina.com.cn/o/z/wuxia/2015-07-09/fxewnie5378673.shtml

### 场景锚点

`anchor_bag_title`

- 类型：OCR + 标题模板
- 用途：识别背包标题

`anchor_bag_grid`

- 类型：模板匹配
- 用途：识别背包格子区

`anchor_sort_button`

- 类型：OCR + 按钮模板
- 用途：识别整理按钮

`anchor_split_button`

- 类型：OCR + 按钮模板
- 用途：识别拆分按钮

### OCR 区域

`roi_bag_title`

- 内容：背包标题

`roi_selected_item_name`

- 内容：当前选中物品名

`roi_selected_item_count`

- 内容：当前选中物品数量

`roi_bag_buttons`

- 内容：整理、拆分等按钮文案

### 可点击目标

`click_bag_slot_n`

- 指定背包格子

`click_sort_button`

- 整理按钮

`click_split_button`

- 拆分按钮

`click_close_bag`

- 关闭背包

### 成功判定

`success_bag_open`

- 背包标题 + 格子区 + 整理按钮至少命中 2 项

`success_item_selected`

- 详情区文字变化或选中高亮出现

`success_sorted`

- 格子排列变化或整理反馈出现

### 失败判定

`fail_grid_missing`

- 未找到背包格子区

`fail_overlay_blocking`

- 被其他弹窗遮挡

`fail_text_unreadable`

- OCR 置信度过低

## ROI 设计原则

在真实实现中，ROI 不应先写死为绝对像素，而应先以“相对窗口百分比”定义，再由运行时转换为像素坐标。

建议原则：

- 标题区：窗口上方 8% 到 18%
- 中央弹窗区：窗口中部 25% 到 75%
- 底部提示区：窗口下方 78% 到 100%
- 右侧任务/提示区：窗口宽度 70% 到 100%

正式校准时，应以真实游戏截图测出更精确的 ROI。

## 点击实现原则

1. 点击目标必须绑定到场景定义，不允许由模型直接输出坐标。
2. 坐标应优先采用“相对窗口坐标”或“相对锚点偏移”。
3. 每次点击前必须确认当前场景锚点仍成立。
4. 每次点击后必须重新截图并判定是否进入预期场景。
5. 如复检失败，不继续盲点，必须回退或暂停。

```json
      "id": "open_first_goods",

    "shop_title && goods_grid && bag_grid"
  ]
}
```

## 样本采集要求

本规格后续必须由真实游戏截图样本校准，且只能使用目标版本、目标设置下的截图进入正式模板库。

统一要求：

1. 固定分辨率。
2. 固定 UI 缩放。
3. 固定语言为简体中文。
4. 固定窗口模式。
5. 不使用直播滤镜。
6. 不使用压缩截图。

### `npc_interact` 采集清单

- 有 `F` 交互提示
- 无 `F` 提示但已选中 NPC
- 右键头像交互菜单
- 不同 NPC 名称样本

### `shop` 采集清单

- 商店刚打开
- 购买页
- 出售页
- 选中商品
- 有确认按钮的购买状态

### `bag` 采集清单

- 空背包
- 满格背包
- 选中单个物品
- 整理按钮清晰可见
- 拆分弹窗出现

## 实现顺序建议

从工程速度看，不建议先做 `npc_interact`，因为它包含 3D 场景、镜头与人物遮挡。

推荐顺序：

1. `bag`
2. `shop`
3. `npc_interact`

理由：

- `bag` 和 `shop` 都是高密度纯 UI 场景，更适合先验证 OCR、模板匹配和点击复检链路。
- `npc_interact` 在识别上更受场景、镜头和人物遮挡影响，适合作为第三步。

## 当前结论

当前阶段不应直接进入“全场景、全自动、全屏自由识别”实现，而应围绕 `npc_interact`、`shop`、`bag` 三个场景建立第一版场景规格、样本库与点击规则。

后续所有真实实现都应以该规格为起点，并由真实目标环境下的截图进行校准。
