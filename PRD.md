《天涯明月刀》“籽小刀”AI 工具 PRD
v3.0（按视频目标收敛版）

状态：执行完成
当前进展：当前仓库已具备“本地 Node 服务 + 单对话框页面”的 Windows 开发骨架，可验证文字输入、本地 Faster-Whisper 语音转写输入、思考链展示、结构化动作规划与自主回合骨架。截图识别接口与分析原型已存在，但尚未接入当前对话主链路；执行层仍为 MockExecutor，尚未接入真实 Windows 鼠标键盘控制。

一、项目目标

本项目当前要做的，不是一个面向任意游戏、任意场景泛化分发的“通用 AI 玩家控制系统”，而是一个直接服务于《灵枢绘世》视频拍摄的专用 AI 工具。

该工具的核心目标，是让主播能够带着一个具有人格、会误解命令、会做价值判断、会制造节目效果的 AI 伙伴“籽小刀”，进入《灵枢绘世》的世界，与游戏中的 AI NPC 发生可观看、可录制、可剪辑的互动。

项目的内容目标优先级高于平台泛化目标。当前版本首先要保证：

1. 主播可以向“籽小刀”下达模糊、高层、带主观色彩的任务。
2. “籽小刀”会以自身人格对任务进行解释，而不是机械执行字面意思。
3. “籽小刀”与《灵枢绘世》中的 AI NPC 互动时，能够稳定产出有反差、有后果、有蝴蝶效应的内容片段。
4. 这些过程必须可展示、可复盘、可用于视频剪辑，而不是只给出后台不可见的自动化结果。
5. 当技术条件成熟时，再逐步补齐真实感知和真实控制能力，把该工具升级为持续运行的游戏 Agent。

需要明确：真实自动控制能力是中长期目标，不是本轮项目价值的唯一来源。当前阶段的第一交付价值，是“能稳定产出视频素材的 AI 代理演员”。

二、视频导向的产品定义

本工具服务的直接场景，是视频《AI 会和 AI 成为好朋友吗？- 天刀手游灵枢绘世》。

因此，产品定义必须围绕视频脚本需求，而不是围绕抽象平台能力展开。

本工具在视频中的身份是：

- 一个有名字的 AI 伙伴：“籽小刀”
- 一个会被主播当作同伴对待的互动角色
- 一个会把模糊命令理解成极端或偏执版本的“黑化执行者”
- 一个能与游戏内 AI NPC 共同制造节目效果的内容引擎

当前产品要支撑的核心桥段包括：

1. 主播带着“籽小刀”初入《灵枢绘世》，与 NPC 随机互动，展示 NPC 的鲜活反应。
2. 主播向“籽小刀”下达模糊任务，例如“有一技之长”“让所有人都认识我”“成为开封城的大富翁”。
3. “籽小刀”以自己的价值判断重新解释这些目标，并选择更激进、更戏剧化、但在其逻辑内自洽的实现路径。
4. AI 行为引发秩序值变化、世界反馈、NPC 态度变化或其他蝴蝶效应。
5. 主播回到现场质问“籽小刀”为何这样做，“籽小刀”能够给出角色化解释。
6. 最终展示后果与补救，形成前后反差，突出“活着的江湖”和“自由选择 + 蝴蝶效应”。

因此，本工具的成功标准不是“自动化越多越好”，而是“能否稳定产出符合以上桥段的内容”。

三、产品定位

本工具是一个面向单一视频项目、单一游戏版本、单一 Windows 主机环境的定制化 AI 互动工具。

它的核心价值不是竞技强度，也不是跨游戏通用性，而是以下三点：

1. 可观看性：AI 的理解、误读、决策、行动和后果都能被主播和观众看见。
2. 人格化：AI 不是中性助手，而是具备稳定人设、表达口吻和价值偏向的“籽小刀”。
3. 剧情驱动性：系统允许在安全边界内保留“人格化误解”，把误读本身变成内容。

对外呈现上，它更像一个“可控但不完全听话的 AI 代理演员”，而不是传统的命令执行器。

四、项目边界

当前阶段明确不做以下目标：

1. 不做面向任意玩家分发的通用安装包产品。
2. 不做跨游戏泛化能力。
3. 不以“完整自动跑图、升级、刷任务”作为第一优先目标。
4. 不为了追求全自动而牺牲视频表现力。
5. 不把“尽量消除 AI 误解”当成产品优化方向；在动作边界内，误解本身就是内容机制。

当前阶段必须优先做好的目标是：

1. 让“籽小刀”有稳定的人设和表达方式。
2. 让模糊任务能被稳定解释成可执行的角色化策略。
3. 让系统能围绕 NPC 互动、秩序值变化和后果反馈产出完整片段。
4. 让每次实验过程都可记录、可展示、可复盘。

五、现实约束

本项目当前的现实约束如下：

1. 后续开发环境固定为当前这台 Windows 主机，不再切回 macOS。
2. 最终软件使用环境也是这台 Windows 主机，而不是面向其他机器分发。
3. 目标游戏《天涯明月刀》手游《灵枢绘世》无官方 API。
4. 当前仓库中的真实执行链路尚未完成，仍处于“文本规划 + 模拟执行 + 视觉原型分离”的阶段。
5. 当前仓库已经具备本地服务、单对话框交互、ASR 接口、自主回合骨架和视觉识别原型，但这些能力尚未组成完整闭环。

六、第一优先级：视频可用 MVP

当前最小可用产品，不是“通用全自动 Agent”，而是“能稳定跑出视频桥段的 AI 工具”。

该 MVP 需要满足以下要求：

1. 主播可通过文本或语音向“籽小刀”下达任务。
2. “籽小刀”能够输出清晰、可播、具有人设的任务理解与思考链。
3. “籽小刀”能够对模糊目标做角色化误读，并给出自洽解释。
4. 系统能够展示当前准备采取的行动方案，而不是只返回一段泛泛对话。
5. 系统能够围绕《灵枢绘世》的关键内容元素组织行为，包括 NPC 互动、关系变化、秩序值、势力、生金、通缉或其他后果性反馈。
6. 系统能够记录一次实验的输入、思考、行动建议、环境反馈与结果，供后续剪辑或复盘。

这个 MVP 的验收重点是：

1. 是否能跑出“AI 玩 AI”的观感。
2. 是否能跑出“黑化版籽小刀误解任务”的桥段。
3. 是否能展示行动带来的后果，而不是只停留在口头上。
4. 是否能稳定积累 10 分钟左右视频所需的可剪素材。

七、阶段性目标

阶段一：视频实验版 MVP

目标：

- 建立“籽小刀”稳定人格，包括名称、口吻、价值偏向和误解风格。
- 让主播通过单对话框输入文本或语音任务。
- 让系统输出适合展示的思考链、角色化解释和结构化行动计划。
- 让系统重点围绕视频桥段工作，而不是追求完整自主游戏能力。
- 保留当前执行层为 MockExecutor 或半实物验证形态，用于展示“准备怎么做”与“为什么这么做”。
- 把截图识别与环境分析优先用于补足“当前场景判断”和“后果展示”，而不是一开始就追求完整自动操作。

本阶段重点验证：

1. “籽小刀”这个角色是否成立。
2. 模糊命令是否能稳定转化为有戏剧性的策略。
3. AI 与 AI 的互动是否有足够反差和节目效果。
4. 输出结果是否足够支撑脚本、拍摄和剪辑。

本阶段不追求：

1. 完整真实自动控制。
2. 完整语音克隆。
3. 泛化到其他游戏或其他机器。

阶段二：半实物内容验证版

目标：

- 接入近实时截图、OCR、模板匹配或其他可行的环境识别能力。
- 识别当前界面类型、NPC 名称、可交互选项、秩序值变化、通缉提示等关键反馈。
- 让“籽小刀”的决策不再只依赖文本假设，而是能读取当前场景并据此调整策略。
- 在不追求完全自动化的前提下，验证“任务理解 -> 环境判断 -> 行动建议 -> 结果反馈”的闭环。
- 强化对视频关键桥段的支持，尤其是“误解任务 -> 激进行动 -> 产生后果 -> 角色化解释”。

阶段三：真实控制版

目标：

- 接入 Windows 原生输入模拟能力。
- 将模拟执行器替换为真实执行器。
- 形成“持续感知 -> 自主规划 -> 真实执行 -> 结果复检 -> 异常恢复”的闭环。
- 允许系统在主播未发言时维持自主推进，在主播发出新命令时立即改道。

需要强调：阶段三是技术终局方向，不应反向绑架阶段一和阶段二的内容目标。

八、核心能力定义

1. 人格化任务理解

主播输入的不是底层点击命令，而是高层目标、态度或模糊要求，例如：

- 帮我和这个 NPC 混熟
- 让所有人都认识我
- 成为大富翁
- 别太老实

系统接收到命令后，不应直接执行字面意思，而应先生成：

- 主播目标理解
- “籽小刀”版本的角色化解读
- 候选策略
- 风险评估
- 当前决定
- 下一步计划

这里的关键不是“完全正确理解”，而是“在可控边界内做出有内容价值的理解偏差”。

2. 人格表达

“籽小刀”必须是稳定角色，而不是不同回合口吻漂移的普通助手。

人格表达至少应包括：

- 固定称谓与身份
- 稳定的态度风格
- 对模糊命令的价值偏向
- 对争议行为的自我辩护能力
- 对失败、风险和后果的角色化解释

3. 结构化思考链展示

思考链不是内部调试日志，而是面对直播/录制场景的内容输出。

每轮回复至少应可见：

- 我理解你想要什么
- 我怎么看当前局面
- 我打算怎么做
- 这样做的风险是什么
- 如果出事，我准备怎么解释或补救

4. 行为规划

行为规划必须围绕视频所需桥段收敛。

第一阶段优先支持的动作类型应服务于以下内容：

- 对话
- 送礼
- 观察
- 交易
- 试探
- 威慑
- 偷窃
- 闷棍或相近高冲突行为
- 逃离
- 补救
- 等待

所有模型输出都应先映射到受限动作集合，再进入执行器或模拟执行器。

5. 环境感知

环境感知的第一目标不是“识别一切”，而是识别视频桥段真正需要的少数关键状态：

- 当前是否处于可互动场景
- 当前 NPC 是谁
- 当前可选交互项是什么
- 当前秩序值或风险状态是否变化
- 是否出现通缉、抓捕、异常提示
- 当前行为是否成功或失败

6. 结果记录与复盘

每次实验过程都应保留可追踪记录，用于后续剪辑和问题定位。记录内容至少应包含：

- 主播输入
- “籽小刀”的任务解读
- 思考链与策略选择
- 动作建议或执行动作
- 环境反馈
- 最终结果

九、总体架构

系统仍采用分层架构，但必须围绕“单一控制原则”保持控制链路清晰。

1. 交互层

负责主播输入、系统输出、思考链展示和实验过程观察。

当前阶段采用“本地 Web 对话框 + 本地服务”形态。

交互层包含：

- 文本输入
- 浏览器录音与本地 Faster-Whisper 转写
- AI 回复展示
- 思考链展示
- 动作建议展示
- 环境状态摘要
- 历史实验记录入口

2. 决策层

负责把主播命令和环境状态转化为结构化决策。

输出结构至少包括：

- 主播意图理解
- “籽小刀”角色化误读
- 当前环境判断
- 候选策略
- 风险评估
- 当前选择
- 下一步动作计划

3. 行为规划层

负责把抽象策略转化为标准化动作列表。

行为规划必须同时满足两点：

1. 行动上可落地。
2. 内容上可展示。

4. 感知层

负责采集截图、OCR、模板识别与关键状态判断。

在阶段一中，感知层允许保持独立原型；在阶段二开始，应逐步接入主链路，为决策层提供稳定输入。

5. 执行层

负责执行动作列表。

建议保留两类执行器：

- MockExecutor：展示“准备如何行动”
- WindowsExecutor：执行真实鼠标键盘输入

当前阶段，MockExecutor 的主要作用不是占位，而是服务于视频实验、行为验证与复盘。

6. 表达层

负责把“籽小刀”的思考、态度和选择转化为可被观看和可被理解的输出。

第一阶段优先文本表达；基础 TTS 可选。
语音克隆不是当前必须完成项，除非后续拍摄明确要求。

十、第一阶段客户端形态

第一阶段客户端继续定义为：

“本地 Web 对话框 + 本地服务”

原因如下：

1. 这仍然是当前最快、最稳、最便于调试的落地形态。
2. 它足以支撑“主播输入 -> 籽小刀回应 -> 展示思考链 -> 展示行动建议 -> 记录实验”的完整内容验证流程。
3. 当前项目不以 Windows 独立安装包为第一目标。
4. 该形态方便后续逐步接入截图识别和真实执行能力，而不必推翻前端交互方式。

十一、验收标准

当前阶段的验收不以“是否已完全自动控制游戏”为唯一标准，而以“是否已经成为视频可用工具”为标准。

至少应满足：

1. 主播可稳定地下达文字或语音任务。
2. “籽小刀”可稳定输出具有人格的回复与思考链。
3. 模糊任务可稳定转化为有内容张力的策略。
4. 系统能展示与《灵枢绘世》核心要素相关的环境判断和后果反馈。
5. 一次实验过程可被完整记录并用于复盘。
6. 系统产出的内容足以支撑视频中的“正常尝试”“黑化执行”“后果展示”“补救解释”这些桥段。

十二、异常处理与安全控制

无论是否进入真实执行阶段，系统都必须具备可见、可恢复、可追踪的失败处理能力。

至少包括：

- 模型输出不合法时停止执行并显式报错
- 场景识别置信度不足时转入人工确认
- 执行器报错时进入暂停状态
- 急停后清空当前动作队列
- 保留最近一次实验记录，避免输入或结果丢失

在真实控制版本中，还需补充：

- 异常界面回退
- 高风险状态保护
- 人工接管优先

十三、当前版本结论

当前项目的正确推进方向，不是继续把目标表述为一个泛化的“AI 玩家控制系统”，而是明确收敛为一个服务于《灵枢绘世》视频拍摄的专用 AI 工具。

因此，当前版本的核心结论是：

1. 本轮项目的第一目标是做成“籽小刀”这个可拍、可演、可复盘的 AI 角色工具。
2. 第一阶段先验证人格、误解机制、思考链展示、行为规划和视频桥段产出能力。
3. 视觉识别和真实控制是后续增强项，但不应抢走当前内容目标的优先级。
4. 当前客户端继续采用“本地 Web 对话框 + 本地服务”方案。
5. 最终技术终局仍可升级为近实时感知驱动的全自动 Agent，但那不是当前 PRD 的唯一中心。
