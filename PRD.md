《灵枢绘世》AI玩家控制系统 PRD
v2.0（按阶段落地版）

状态：执行到一半

一、项目目标

本系统的目标，是构建一个“可被观看的 AI 实体玩家”。
它不仅要能够理解主播的指令、形成可展示的思考过程、输出带人格的表达，还要在条件成熟时进一步接入真实游戏控制能力，完成 NPC 互动、社会行为与剧情推进。

本项目当前的现实约束是：

1. 现阶段主要开发环境为 macOS。
2. 目标游戏无官方 API。
3. 类似按键精灵或 Windows 底层输入模拟的能力，当前不适合在 macOS 上作为第一阶段核心交付。

本项目的最终交付要求是：

1. 成品必须提供 Windows 安装包，交付形态以 `.exe` 安装程序为准。
2. 目标用户应当能够在普通 Windows 机器上点击安装并直接运行，不需要额外安装 Node、Python、浏览器运行时或其他依赖环境。
3. 因此，开发过程中应尽量减少对外部环境和手工配置的依赖，所有必要运行组件应在最终安装包内完成封装。

因此，本项目不再以“第一天就完成真实自动操控”为目标，而采用分阶段建设策略：

第一阶段，先完成“AI 可被观看、可被指挥、可展示思考、可输出人格”的核心体验。
第二阶段，补齐视觉识别与半自动验证能力。
第三阶段，再接入 Windows 平台下的真实输入控制能力，形成完整闭环。

二、产品定位

本系统是一个面向直播和内容创作场景的 AI 互动控制系统。
它的核心价值不是竞技强度，而是“可观看性”与“人格化”。

对外呈现上，它应该像一个拥有独立性格、会误解、会权衡、会解释自己行为的 AI 玩家。
对内实现上，它应该是一个可拆分的系统：

- 平台无关部分：语义理解、思考链生成、行为规划、日志、状态管理、前端展示、人格表达。
- 平台相关部分：屏幕采集、OCR、模板识别、鼠标键盘注入、系统级音频或输入控制。

三、阶段性目标

阶段一：macOS 可运行的最小产品（MVP）

目标：

- 让主播可以通过文字方式向 AI 下达指令。
- 让 AI 生成结构化思考链，并在客户端实时展示。
- 让 AI 输出行为规划结果，而不是直接执行真实鼠标点击。
- 让客户端具备开始、暂停、急停、日志查看等控制能力。
- 让系统的“人格表现”和“直播观感”先被验证。

本阶段不追求：

- 真实控制游戏。
- 完整语音克隆。
- 高精度 OCR 或跨场景泛化。

阶段二：视觉识别与半实物验证

目标：

- 接入截图上传、实时截屏或录屏帧输入。
- 识别当前界面类型、NPC 名称、基础互动选项、异常提示。
- 验证“语义理解 -> 视觉判断 -> 行为规划”的链路是否稳定。
- 执行层依然允许保留为“模拟执行器”，只输出下一步动作建议。

阶段三：Windows 真实控制版本

目标：

- 接入 Windows 原生输入模拟能力。
- 实现鼠标、键盘、行为节奏随机化与中断控制。
- 将模拟执行器切换为真实执行器。
- 形成“识别 -> 规划 -> 执行 -> 反馈 -> 恢复”的完整自动化闭环。

四、总体架构

系统采用分层架构，而非单体脚本式实现。

1. 交互层

负责接收主播输入，并将系统状态反馈给主播。
第一阶段建议以 Web 控制台作为客户端形态，即一个本地运行的 HTML 页面。

交互层包含：

- 文字输入
- 后续可扩展的语音输入入口
- 思考链展示
- 当前行为状态展示
- 历史日志展示
- 开始、暂停、急停控制

2. 决策层

负责把自然语言指令转化为结构化任务。
该层由大模型驱动，但输出必须是受控结构，而不是自由散文。

建议输出结构至少包括：

- 主播意图理解
- 当前环境判断
- 候选策略
- 风险评估
- 当前选择
- 下一步动作计划

3. 行为规划层

负责把抽象目标转化为可执行步骤。
例如，“和 NPC 混熟”不会直接变成点击，而应先变成：

- 确认当前是否已处于对话范围
- 判断 NPC 是否可互动
- 判断背包内是否有礼物
- 若无礼物，转为闲聊或其他互动路径
- 输出下一步操作建议

这一层的输出应该是标准化 Action 列表，而不是自然语言段落。

4. 感知层

负责理解当前游戏或模拟环境的状态。
这一层不应和具体平台强耦合，而应通过适配器接入。

感知层输入形式可以按阶段演进：

- 阶段一：手动选择当前场景，或手动上传截图
- 阶段二：接入实时截图、OCR 和模板匹配
- 阶段三：接入稳定的游戏窗口采集方案

5. 执行层

负责执行 Action 列表。
该层必须抽象为接口，以便不同平台替换实现。

建议至少有两种执行器：

- MockExecutor：只展示“将要执行什么”，不真实点击
- WindowsExecutor：在 Windows 环境下执行真实鼠标键盘输入

6. 表达层

负责把 AI 的思考、态度和选择转化为对外可感知的表达。
第一阶段可先用文本展示与普通 TTS。
后续再升级为定制语音人格或语音克隆方案。

五、第一阶段客户端形态

第一阶段客户端不再定义为 Windows 独立程序，而定义为：

“本地 Web 控制台 + 本地服务”

其中：

- Web 控制台负责展示和交互，可以是一个 HTML 页面。
- 本地服务负责调用模型、维护状态、生成思考链和行为计划。

这样设计的原因是：

1. 开发速度快，适合快速迭代交互体验。
2. 页面结构天然适合展示思考链、日志和状态流。
3. 后续若需要桌面化，可再封装为 Electron、Tauri 或原生壳。
4. 前端与执行层解耦，后期切换到 Windows 时无需重写大部分业务逻辑。

六、最终交付形态与安装要求

最终面向用户的正式版本，不以源码仓库、命令行启动或手动配置环境的方式交付，而应以 Windows 安装包交付。

具体要求如下：

1. 安装产物应为标准 Windows 可执行安装程序，优先考虑 `.exe` 安装包。
2. 用户安装完成后，应可通过桌面图标或开始菜单直接启动，不要求额外打开终端或执行命令。
3. 安装过程应默认包含应用运行所需的全部组件，不要求用户额外安装 Node、Python、OCR 运行时或其他通用依赖。
4. 若系统需要本地服务、模型调用配置、截图能力或输入控制模块，应在安装后由程序自行管理，而不是要求用户自行拼装运行环境。
5. 第一阶段可以继续使用“本地 Web 控制台 + 本地服务”架构验证功能，但后续进入 Windows 正式交付阶段时，应将其封装为一体化桌面安装程序。
6. 技术选型时应优先考虑未来易于封装为 `.exe` 的方案，避免引入强依赖当前 macOS 开发环境的实现。

七、第一阶段核心功能

1. 指令输入

主播通过文本输入指令，例如：

- 帮我和这个 NPC 混熟
- 赚钱
- 给他点颜色看看
- 别太老实

系统接收后，进入一轮新的任务状态。

2. 思考链展示

系统生成受控、可读、适合直播展示的思考链。
思考链不是无限展开，而应控制长度、节奏和风格。

建议思考链分为三段：

- 我理解你想要什么
- 我判断当前环境是什么
- 我准备怎么做

示例：

“目标不是单次互动，而是建立关系优势。”
“当前最重要的是先确认对方是否能进入高频互动状态。”
“如果资源不足，我会先走低成本接近策略。”

3. 行为状态区

用于显示结构化状态，而非纯文案。
建议展示内容包括：

- 当前目标
- 当前场景
- 候选策略
- 已选策略
- 风险等级
- 下一步动作

4. 日志区

记录每一轮行为的关键节点，例如：

- 收到指令
- 完成意图解析
- 更新场景判断
- 生成行为计划
- 执行器返回结果
- 进入暂停或异常状态

5. 控制区

至少包含以下按钮：

- 开始
- 暂停
- 继续
- 急停
- 清空上下文

其中“急停”必须具备最高优先级，用于直播场景下的人工接管。

6. 模拟感知输入

在没有真实游戏控制能力时，系统仍然需要环境输入。
第一阶段可以采用以下任一方案：

- 手动选择当前界面状态，例如“对话 / 背包 / 交易 / 牢房”
- 上传游戏截图供模型或 OCR 分析
- 使用预设测试场景数据进行联调

7. 模拟执行器

第一阶段不执行真实点击，只输出诸如以下内容：

- 打开互动菜单
- 选择“交谈”
- 检查是否存在“赠礼”选项
- 若存在，则优先选择低风险礼物

这部分既用于调试，也用于验证行为规划质量。

八、语义理解与行为规划要求

语义层必须允许一定的“人格化偏差”，即 AI 可以有风格化解读，但不能脱离动作边界。

例如：

- “帮我混熟”可以被解释为高频互动、送礼或制造强记忆点。
- “给他点颜色看看”可以被解释为威慑、试探、挑衅，但不能无限制发散。

为保证可控性，建议建立受限动作集合。

动作集合示例：

- talk
- gift
- inspect
- trade
- threaten
- steal
- strike
- escape
- wait

所有模型输出都应先映射到受限动作集合，再进入执行器。

九、视觉识别设计

视觉识别模块目标不是“一开始就全自动”，而是逐步提高自动化程度。

第一阶段：

- 允许人工提供场景信息
- 支持上传截图进行离线分析

第二阶段：

- OCR 识别界面文字
- 模板匹配识别固定 UI
- 提取 NPC 名称、菜单选项、秩序提示、通缉提示等信息

第三阶段：

- 与实时采集方案整合
- 为执行层提供稳定、低延迟的状态反馈

十、语音与人格模块

人格模块是本系统的内容价值核心之一。
但在工程排期上，应按复杂度分层处理。

第一阶段：

- 文本人格先行
- 可选基础 TTS 播报

第二阶段：

- 优化人设口吻
- 增强思考链节奏与反应速度

第三阶段：

- 接入特定人格语音方案
- 实现更稳定的角色化表达

十一、异常处理与安全控制

无论是否进入真实执行阶段，系统都必须具备中断与恢复机制。

至少包括：

- 模型输出不合法时停止执行
- 场景识别置信度不足时转人工确认
- 执行器报错时进入暂停状态
- 急停后清空当前动作队列

在真实控制版本中，还需补充：

- 异常界面回退
- 通缉或高风险状态保护
- 人工接管优先

十二、推荐技术实现方向

第一阶段推荐架构：

- 前端：HTML + CSS + JavaScript，或基于轻量前端框架
- 后端：本地服务，负责模型调用、状态机、日志、计划生成
- 通信：HTTP + WebSocket 或 SSE，用于实时推送思考链和状态
- 感知层：截图上传接口 / 测试场景输入接口
- 执行层：MockExecutor

第二阶段新增：

- OCR 服务
- 模板匹配模块
- 截图采集模块

第三阶段新增：

- Windows 输入模拟模块
- 真实执行器
- 更完整的异常恢复机制

补充约束：

- 正式交付链路必须预留桌面封装方案，不能停留在“需要用户手动运行 Node 服务”的形态。
- 如需使用 Web 技术栈，应优先选择后续容易打包为 Windows 安装程序的封装方式。
- 配置项可以保留为可编辑文件或安装后设置项，但不应要求用户先自行搭建开发环境。

十三、当前版本结论

本项目当前最合理的推进路径，不是直接追求“按键精灵式全自动控制”，而是先完成一个可运行、可展示、可调试、可验证产品价值的 AI 控制台。

因此，当前版本的核心结论是：

1. 客户端可以是一个 HTML 页面。
2. 更准确地说，第一阶段应采用“本地 Web 控制台 + 本地服务”方案。
3. 真实游戏控制能力不应阻塞第一阶段开发。
4. 第一阶段先验证交互体验、人格表现、思考链展示和行为规划质量。
5. 后续在 Windows 环境下补齐执行器，即可升级为完整控制系统。
6. 最终正式交付必须收敛为“普通 Windows 用户可直接安装和启动”的 `.exe` 安装包形态。
