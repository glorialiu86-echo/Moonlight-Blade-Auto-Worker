# Windows 主机部署与运行方案

状态：未执行

## 目标

本方案用于替代此前“Windows `.exe` 安装包交付”的约束，明确系统后续的真实目标是：

- 在一台固定的 Windows 主机上部署
- 在该主机上稳定运行《灵枢绘世》AI 玩家控制系统
- 优先保证识别速度、点击稳定性、调试便利性

本方案不以通用安装包分发为优先目标。

## 新约束

1. 运行环境固定为一台专用 Windows 主机。
2. 允许在该主机上安装必要依赖。
3. 允许保留命令行启动、本地服务启动、浏览器控制台等开发态运行方式。
4. 项目优先目标从“封装与分发”转为“识别、决策、点击链路尽快跑稳”。

## 总体架构

系统仍分为两部分，但以 Windows 主机为中心部署：

### 1. 控制台层

保留现有 `Node 服务 + HTML 控制台` 结构。

作用：

- 接收主播文字指令
- 后续可扩展语音输入
- 展示思考链、场景状态、日志与动作计划
- 显示截图 OCR / 场景识别结果

运行形态：

- Windows 主机本地启动服务
- 浏览器访问本地页面

### 2. Windows Agent

新增一层面向真实控制的 Windows Agent。

作用：

- 获取游戏窗口截图
- 执行模板匹配与局部 OCR
- 调用千问补充视觉判断
- 执行真实鼠标键盘输入
- 点击后复检

与控制台通信方式：

- 本地 HTTP
- 或者本地 IPC

当前建议优先使用本地 HTTP，便于调试和解耦。

## 为什么不再优先做桌面封装

因为当前项目是定制化自动化系统，不是面向不特定用户分发的软件产品。

在这种约束下，最重要的是：

- 那台固定 Windows 主机能否稳定识别 UI
- 能否在固定窗口环境下高成功率点击
- 是否容易排查识别和点击失败原因

相比之下，是否封装为安装包已经不是当前关键路径。

## 推荐技术路线

### 控制台与决策层

保持现有技术栈：

- Node 服务
- HTML/CSS/JavaScript 控制台
- 千问文本模型做规划
- 千问 Vision/OCR 做补充视觉理解

### Windows 识别层

主链路采用：

- 模板匹配
- 局部 OCR
- 少量规则判断

大模型只做：

- OCR 结果补充解释
- 歧义判断
- 场景总结
- 高层策略推断

不做：

- 全屏自由定位点击
- 逐帧把截图完全交给模型决定坐标

### Windows 点击层

执行规则：

1. 先识别当前场景
2. 再在场景定义文件里找到可点击目标
3. 使用相对窗口坐标或相对锚点偏移点击
4. 点击后立即截图复检
5. 失败则停止或回退

## 阶段性推进顺序

### 阶段 A：固定环境校准

在 Windows 主机上固定以下条件：

- 分辨率
- UI 缩放
- 游戏语言
- 窗口模式
- 游戏版本

输出：

- 环境基线文档
- 第一批截图样本

### 阶段 B：纯 UI 场景识别

优先实现：

1. `bag`
2. `shop`

原因：

- 它们是纯 UI 场景，稳定性高
- 识别与点击都比 3D 场景简单
- 适合先打通 OCR、模板匹配和复检链路

输出：

- 背包场景识别成功
- 商店场景识别成功
- 关键按钮点击成功

### 阶段 C：NPC 交互

实现：

- 目标选中
- `F` 交互提示识别
- 对话前状态判定

原因：

- 该场景涉及 3D 画面、镜头变化和人物遮挡
- 适合作为第三步而不是第一步

### 阶段 D：闭环验证

串起：

- 截图
- 识别
- 规划
- 点击
- 复检

输出：

- 第一条真实场景自动化链路

## 建议优先打通的第一条闭环

优先场景：

- `shop`

原因：

- 商店和背包天然联动
- 界面元素比 NPC 对话更稳定
- OCR 和模板匹配更容易先见效

推荐闭环：

1. 打开商店
2. 识别商店标题
3. 识别商品区与背包区
4. 读取商品名/价格
5. 决策层输出“先对话 / 先观察 / 先买入”
6. 点击一个确定目标
7. 复检是否进入下一状态

## 运行方式建议

在 Windows 主机上，短期建议采用以下方式运行：

1. 本地启动 Node 控制台服务
2. 本地启动 Windows Agent
3. 浏览器打开控制台页面
4. 游戏保持固定窗口模式运行

这种方式的优点是：

- 调试速度快
- 错误定位清晰
- 识别链和点击链分离
- 不需要先解决桌面封装问题

## 部署要求

虽然不再做安装包，但仍需要形成明确部署清单。

部署清单至少应包括：

- Node 版本
- 依赖安装命令
- 环境变量说明
- Windows Agent 启动方式
- 游戏端固定设置要求
- 截图样本目录结构

## 当前结论

在“固定 Windows 主机运行”的新约束下，项目后续的最优目标不再是安装包交付，而是尽快把以下链路跑稳：

- 固定环境
- 场景识别
- OCR
- 决策规划
- 真实点击
- 点击后复检

这将明显快于继续围绕桌面封装和通用分发做工程设计。
